# Beat the House — 0.3.2 Low-End & Web Cleanup Board

Created: 2026-07-04
Owner: project management
Scope: 0.3.2 is the main cleanup pass of the v0.3 set. NO new content, NO new
features. The mission is exhaustive: identify and review EVERY function in the
production codebase for lag, per-frame cost, memory that is allocated and never
released, allocation churn, and anything else that makes the game run slowly on
a low-end system or in the web export — then fix what is found.

How this board differs from 0.3.1: the 0.3.1 Performance & Stability Audit was
symptom-driven (chase the known hot spots, fence them with budgets). This board
is coverage-driven: a function-level census of the whole repository where every
production function gets classified, every hot-class function gets read and
costed, and the low-end/web target — never measured in 0.3.1 — becomes the
budget baseline. Nothing is exempt because "it was fine on the dev box".

Known leads this board exists to run down (verify and extend, do not
rediscover):

- Every 0.3.1 performance number was measured headless or on a 4070 Ti
  desktop. The web export (`builds/itch/BeatTheHouse-web.zip`, 16.3 MB) and
  low-end hardware have NEVER been profiled. Web is single-threaded WASM with
  slower allocation, slower JSON, and a stricter memory ceiling; the SC.2
  budget table has no web or min-spec column.
- SC.3's final soak recorded a POSITIVE memory slope of 719,852 bytes/sample
  (~4.3 MB/simulated hour) where SA.4's soak had measured -455,080 bytes
  post-warmup. The cap assert passed; the slope reversal was never explained.
- The pinball feature's structural cost is documented and unfixed:
  docs/plans/pinball_feature_rework_plan.md §1 records per-tick
  read_machine -> duplicate(true) -> write_machine round-tripping of the whole
  machine dict including the live session, O(history) copy growth, Variant
  dictionary math, and per-step TableScript.new(). 0.3.1's SB.1 explicitly
  made "no edits inside pinball internals"; `pinball_feature.gd` still
  contains 7 `duplicate(` calls today.
- docs/plans/dead_code_audit_report.md (2026-07-01) is a READ-ONLY audit with
  a prescribed removal sequence that was only partially executed (R9 removed
  speculative_content.md). Dead scripts, dead data keys, and unreferenced
  assets still ship in the export.
- The audio stack — `procedural_music_player.gd` (3,542 lines),
  `sfx_player.gd` (1,904 lines), `web_audio_bridge.gd` — was reworked for web
  playback late in 0.3.1 (commit `6ae682b`) and has never had a performance
  or allocation audit. Procedural music synthesizes/schedules continuously,
  which is exactly the kind of steady per-frame cost a low-end machine feels.
- Roulette's enforced resolve budget is 16/18/20 ms avg/p95/max on the dev
  box — already more than a whole frame at 60 FPS. On a low-end machine or
  web that is a multi-frame hitch on every spin.
- Size concentration: `foundation_main.gd` 11,154 lines, `blackjack.gd`
  5,684, `run_state.gd` 4,083, `pixel_scene_canvas.gd` 3,843. The census must
  not skip the big files because they are big.

## Low-end target definition

All budgets on this board are set against two targets, not the dev box:

- **MIN-SPEC DESKTOP**: 2015-class dual-core laptop CPU with integrated
  graphics. Approximated on the dev box by the measured low-end scale factor
  from L0.2 (default working assumption until measured: 4x GDScript cost,
  budget = dev-box budget / 4 headroom check).
- **WEB**: the itch.io HTML5 export running in current Chrome and Firefox on
  the same class of hardware. Measured directly via the L0.2 web harness, not
  approximated.

A change "passes" only if the dev-box numbers leave headroom for the scale
factor, or the web harness measures it directly under budget.
