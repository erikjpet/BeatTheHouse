# 0.5 Onboarding Plan — The First Card Tutorial + the Coach System

Status: PLANNED, owner-approved direction (2026-07-14). Depends on the
0.5 Grand Casino rework landing first (`docs/plans/
0.5_grand_casino_rework_plan.md`) — the tutorial's payoff mints the
player's first Players Card, which is that plan's slice-8 item. Executes
AFTER the gc05 and audio queues, BEFORE the 0.5 UI overhaul (the overhaul
must skin these surfaces, not retrofit them).

## Owner direction (locked)

- A tutorial run ALWAYS starts on a fresh profile ("new file"): a guided
  level that explains the game and hands the player an achievable easy
  win, ending with their FIRST Players Card so every new player starts
  the real game holding one.
- Beyond the one-time level, build a DYNAMIC system that explains how to
  play — reusable, data-driven, not hardcoded script.

## Part 1 — The Coach system (the reusable engine)

A data-driven contextual instruction engine, used by the tutorial run
and by first-time tips in normal runs forever after.

### Data model — `data/tutorial/lessons.json`

Each lesson step declares:

- `id` — stable, one-shot per profile.
- `trigger` — context conditions: screen, environment kind/archetype,
  game id, state predicates (first debt taken, heat crossed N, closing
  warning shown, chips gained, card tier reached…), and ordering
  dependencies on other lesson ids.
- `anchor` — what it points at: an interactable object id, a HUD
  element key, a surface action, or none (center card).
- `copy` — short player-facing text (content_style_guide voice: plain,
  second person, one idea per step).
- `completion` — what dismisses it: the anchored action performed, any
  action, or explicit OK.
- `gating` — optional: while active, restrict other interactables
  (tutorial uses this; normal-run tips NEVER gate).

### Presentation

- Non-blocking coach bubbles anchored to canvas objects/HUD regions,
  reusing the interaction-focus rect system for highlight + dim, and
  talk-dock/panel visual idioms. One bubble at a time, queued.
- A distinct visual identity (the "dealer's advice" voice) so coaching
  reads apart from events/dialogue.
- Built as an extracted component (`CoachOverlay` + pure view model);
  foundation_main gains wiring only.

### First-time tips (normal runs)

The same engine fires one-shot contextual tips outside the tutorial:
first heat spike, first debt, first closing-time eviction, first pawn,
first chips at the Grand Casino, first card tier, first prestige carry.
Seen-state persists in the PROFILE (schema-versioned persistence
exists), so tips never repeat across runs. A settings toggle disables
all tips ("Coach: on/off") and a "reset tips" option restores them.

### Engine rules

- Triggers evaluate at action boundaries only; no wall-clock; zero
  per-frame allocation (bubble view built on state change).
- The engine never mutates simulation state — it observes flags the
  simulation already sets.
- Fully data-driven: adding a lesson is a JSON edit.

## Part 2 — The First Card tutorial run

### Entry

- A fresh profile (no run history, no collection, `tutorial_completed`
  unset) auto-starts the tutorial when the player begins their first
  run. Prominent but not preachy: "First night in town" framing.
- Skippable at any time (confirm dialog: "skip the lessons?") — the run
  converts in place to a normal run or exits to menu; skipping marks
  `tutorial_completed`. Replayable later from the start screen
  ("Replay the lessons").

### Structure — a compressed authored arc (fixed tutorial seed + a
`tutorial_first_card` challenge config)

1. **Home**: walk, inspect a container, see the (empty) loadout — the
   between-runs loop in 60 seconds. Leave via the map (map lesson:
   hours visible on venues).
2. **Corner store**: buy one cheap item with starting cash (money +
   items lesson).
3. **Bar**: guided blackjack at low stakes — wager, actions, outcome
   presentation; heat introduced by WATCHING it move, not suffering it.
   The count-challenge gets a one-step taste ("the count is showing —
   peek if you dare") with a safety note.
4. **Talk/event**: one scripted friendly event through the talk dock
   (choices lesson).
5. **The invitation**: a scripted tutorial event grants the Grand
   Casino invite directly (teaching the invite gate exists, without
   the full tier-2 climb).
6. **Grand Casino, easy mode**: tutorial-config objective overrides
   (tiny net-winnings target, 1-2 games minimum, generous heat
   ceiling — all in the challenge config, never global). Guided to the
   high-limit... no: MAIN FLOOR only for the tutorial. Play, win small,
   meet chips, cash out at the Cage, meet LINDA — she walks the player
   through the card review and issues the card.
7. **Victory + the hook**: run report shows the win; the first Players
   Card mints into the collection (gc05 slice-8 machinery). Final coach
   beat back home: "Carry it for recognition — lose a run holding it
   and it's gone forever." The tutorial's last lesson is the game's
   core tension.

### Easy-win mechanism (honest, deterministic)

Prefer a hand-picked tutorial SEED whose scripted path wins naturally,
plus the generous tutorial objective overrides — over code-forced
outcomes. If seed auditioning cannot guarantee the beats, allow
tutorial-challenge-scoped friendly modifiers (data-driven, clearly
scoped to the tutorial challenge id). Never touch normal-run RNG.

### State

- Profile: `tutorial_completed`, `tips_seen{}` (both schema-versioned).
- The tutorial run itself is a real run (saves, determinism, gates all
  apply); its challenge config id excludes it from daily/challenge
  stats and leaderboard-relevant history.

## Engineering constraints (binding)

Determinism (fixed seed, action-boundary triggers), zero-copy per-frame,
idle-liveness untouched, save/profile schema compatibility, extracted
components only, suite timeout = max(300s, baseline×1.5). A fresh
profile MUST reach the card in under ~15 minutes of guided play; the
skip path must leave zero coach residue.
