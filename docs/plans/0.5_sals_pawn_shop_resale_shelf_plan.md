# 0.5 Sal's Pawn Shop Rework — Persistent Resale Shelf and Float-Rarity Tutorial

Status: PLANNED, owner-directed (2026-07-20).

This plan expands the landed meta-home version of Sal's Pawn Shop from a
sell-only counter into a persistent six-slot resale shop. Sal independently
stocks one newly generated collection item after every successful run, sells
those exact instances for meta gold, comments on purchases and sales, and uses a
deliberately suspicious starter bargain to teach the player how rare float
values change an item's price.

This is a rework of landed systems described by
`docs/plans/item_collection_meta_system_plan.md` and the executed v0.4 meta-home
prompts. This plan supersedes their pawn-value assumptions where they conflict:
collection-item pawn value now follows the edge-float rarity curve defined here.
Bag pricing, Grand Casino chip-stack fencing, Players Card rules, housing costs,
trade-ups, and run-side item effects remain unchanged unless explicitly named.

## Locked owner decisions

1. Sal has exactly **six persistent resale shelf slots**.
2. A fresh profile begins with exactly **one starter shelf item**.
3. Every successful run stocks exactly **one new independent item** for Sal.
   This includes normal, tutorial, daily, challenge, and any other successful
   run, even when that run is otherwise ineligible for player meta-collection
   rewards.
4. Sal's stocked item is unrelated to the player's end-of-run bag or selected
   reward. Player sales also never become Sal stock.
5. Stock generation uses the same item-definition and four-float roll machinery
   as opening a collection bag. The system first chooses a virtual bag by the
   normal collection/tier logic, then rolls one eligible item from that bag.
6. Empty shelf slots fill before replacement. Once all six slots are occupied,
   a successful run replaces one eligible occupied slot chosen at random.
7. Stock, exact item instances, floats, slot positions, listing modes, prices,
   and generation state persist across sessions.
8. Normal shelf asking price is **1.5× the exact amount Sal would pay for that
   same instance**, rounded up to whole meta gold.
9. The starter item is a deliberate exception: its initial asking price is
   **0.75× its exact Sal pawn quote**. It is therefore sold below what Sal would
   ordinarily pay for it.
10. The starter item has low durability (`condition < 0.30`) and exactly one
    seeded non-durability float (`potency`, `resonance`, or `usage`) forced to
    `0.01` or lower. The rare float is not `condition`, because low durability is
    bad rather than valuable.
11. Immediately after the player first buys that starter item, Sal notices the
    rare edge float and offers to buy the exact instance back for **1.25× his
    normal pawn quote**, rounded up.
12. If accepted, Sal pays the player, recovers the exact instance, returns it to
    the just-vacated shelf slot, and relists it for **10× his normal pawn quote**,
    rounded up. He mocks the player through authored dialogue. This special
    buyback never triggers again for that instance.
13. If the player refuses, they keep the item and the special offer is resolved
    permanently.
14. Sal has authored dialogue/reactions for purchases and sales, with the
    starter purchase/buyback/relist as the first signature interaction defining
    his opportunistic tone.
15. Float rarity affects the pawn value of all ordinary collection-item
    instances. Bags and special non-collection item classes keep their existing
    pricing policies.

Shelf instances are not player-owned and must not appear in `owned_instances`,

`next_instance_id` normalization must include instances currently on Sal's shelf
and in any pending buyback transaction.

### Fresh profile and migration

- A brand-new store contains six normalized slots with exactly one starter
  listing and five empty slots.
- Existing profiles receive the same one-time shelf initialization on load,
  regardless of housing tier or prior Sal visits.
- Migration never changes owned items, bags, gold, housing, loadout, trade-up or
  sale history.
- Unknown store fields remain preserved.
- Repeated normalization/load/save never creates another starter item.
- The unpurchased starter listing is protected from random full-shelf
  replacement until its first purchase, guaranteeing that the tutorial remains
  available. Once purchased or its offer is resolved, it follows ordinary slot
  replacement rules. A 10× mocking relist may later be randomly replaced like
  any other occupied listing.

## Generating independent stock

### Virtual-bag pipeline

Extract or reuse one shared, testable generation seam rather than copying the
bag-opening rules:

1. Choose a collection using the same collection-selection rule used for normal
   run-end bag generation.
2. Choose a tier using that collection's standard `drop_table` weights.
3. Resolve the bag definition for that collection/tier.
4. Obtain candidates through
   `CollectionItemResolver.bag_item_options_for_bag(...)`.
5. Choose one candidate with the stock stream.
6. Roll its four floats through the same deterministic
   `CollectionItemResolver.roll_instance(...)` path used by bag opening.
7. Normalize it and assign Sal-specific provenance without granting a player bag
   or player-owned instance.

Regular Sal stock must have exactly the same item/float distribution that a
normal virtual bag opening would produce. Do not link its identity, collection,
tier, definition, or seed to the reward bag offered to or selected by the
player. Prestige/drop bonuses tied to a player's reward do not affect Sal's
independent roll unless they are part of the shared base virtual-bag rule.

Use a named, persisted meta RNG stream. UI opens, room entry, redraw, view-model
builds, quote requests, and save/load must not roll or replace stock.

### Starter override

Generate the starter through the same virtual-bag pipeline, then apply only
these explicit tutorial overrides before final normalization:

- clamp/set `condition` to a deterministic value strictly below `0.30` and above
  zero;
- choose one of `potency`, `resonance`, or `usage` from the named starter stream
  and set it to exactly `0.01` (or a deterministic value in `[0, 0.01]`);
- record the chosen rare float name/value in provenance so dialogue and tests do
  not rediscover it heuristically;
- mark listing mode `starter_discount` and tutorial eligibility.

All other identity and float values remain the result of the normal roll.

## Successful-run stocking

Every transition to a successful terminal run (`RUN_STATUS_ENDED`, not failed)
must attempt exactly one Sal stock insertion. This shelf mutation occurs even if
`meta_collection_enabled_for_run()` is false. It is independent of whether the
player receives, selects, declines, or opens a bag.

The terminal bridge must:

1. establish/reuse a stable serialized completion receipt unique to that actual
   run attempt;
2. check both run-side and profile-side idempotency before generating;
3. generate exactly one independent stock item;
4. insert it into an empty slot, preferring stable lowest slot index;
5. when no empty slot exists, choose one eligible occupied slot uniformly from
   the persistent named stream and replace it;
6. never choose a protected unpurchased starter slot;
7. record the replaced listing for bounded audit/history, without granting it to
   the player or paying anyone;
8. persist the shelf, RNG, and receipt atomically;
9. mark the terminal run as processed and expose a concise run-report line.

Reopening a run report, reloading a terminal save, retrying profile save, or
re-entering Sal's shop must never stock a second item for the same successful
run. Replaying the same seed as a genuinely new run must still stock a new item;
therefore seed text alone is not a sufficient receipt.

## Float-rarity pawn valuation

### Terminology

The ordinary collection instance contains four floats in `[0,1]`:

- `potency` — symmetric rarity: values near 0 or 1 are rare;
- `condition` — durability for this pricing rule: only values near 1 are
  valuable, because low durability is bad;
- `resonance` — symmetric rarity;
- `usage` — symmetric rarity for market valuation, even though it retains its
  landed gameplay semantics.

### Locked curve

For `x` in `[0,1]`:

```text
edge(x)       = abs(2x - 1)^4
durability(c) = c^4

rarity_multiplier =
    1
    + 0.5 × edge(potency)
    + 0.5 × edge(resonance)
    + 0.5 × edge(usage)
    + 0.5 × durability(condition)
```

The result ranges from 1× to 3×. The fourth power makes increases exponential
near the rare edges. A 50% potency/resonance/usage float adds zero rarity value;
1% and 99% add nearly the maximum contribution. Condition differs: 1% adds
almost nothing while 100% adds the full durability contribution.

### Quote calculation

For ordinary `ITEM_CLASS_COLLECTION` items:

```text
pawn_quote = max(1, round(tier_base_sale_value × rarity_multiplier))
normal_resale_price = max(pawn_quote + 1, ceil(pawn_quote × 1.5))
starter_price = max(1, round(pawn_quote × 0.75))
starter_buyback_offer = max(1, ceil(pawn_quote × 1.25))
mocking_relist_price = max(1, ceil(pawn_quote × 10.0))
```

The starter price is the only intentional case that can be below Sal's pawn
quote. The 10× relist and the 1.25× offer use the exact same instance and the
same updated pawn quote. Normal player sales and all shelf listings call one
pure quote evaluator so previews and committed transactions cannot disagree.

This new rarity curve replaces conflicting legacy collection-item pawn-value
multipliers for Sal; do not secretly multiply it by the old condition-band/
spent-value curve and thereby make a 1% symmetric float less valuable than a
50% float. Retain old multipliers wherever they still serve run effects or
non-Sal presentation. Bag sale prices and special item-class policies such as
Grand Casino chip-stack face-value fencing are unchanged.

Expose a price breakdown containing tier base, all four float contributions,
rarity multiplier, pawn quote, listing policy multiplier, and final asking
price. Use the same breakdown in UI, dialogue, tests, and history.

## Purchasing shelf items

- Purchases use meta gold only.
- Each occupied slot is a physical, focusable shop object at one of the six
  existing pawn-shop `item_spots`.
- Inspecting an item shows collection, tier, exact four float percentages,
  durability, rarity contribution/badges, Sal's pawn quote, asking price, and
  listing status.
- Confirm purchase before mutating.
- Validate gold and player storage capacity first.
- A normal successful purchase atomically debits gold, removes the exact listing
  from its slot, and transfers the same instance ID/floats/provenance into
  `owned_instances`.
- A failed purchase mutates neither gold nor shelf.
- Purchased slots remain empty until a later successful run fills them. Buying
  does not immediately reroll stock.
- Save, refresh the physical shelf, and present Sal's purchase dialogue only
  after a successful transaction.

Player items sold through the existing sell counter are destroyed for gold as
before. They are never copied, moved, or rerolled onto the shelf. Sale history
may reference the transaction but cannot be used as stock input.

## Sal dialogue and the starter tutorial

### General reactions

Give Sal a stable speaker identity and add authored transaction reactions through
the existing dialogue/talk system:

- short purchase acknowledgements;
- short sell/pawn acknowledgements;
- distinct refusal/insufficient-gold copy where appropriate;
- the signature starter sequence below.

Keep routine lines concise and avoid interrupting every browse/hover. Trigger
them only on committed buy/sell transactions. Selection from any reaction pool
must be deterministic at action boundaries and persist enough state to avoid
rerolling on UI refresh.

### Signature starter sequence

On the first successful purchase of the protected starter listing:

1. Complete and save the purchase normally; the player owns the exact item.
2. Immediately open a blocking Sal dialogue. He names the rare float and its
   displayed percentage, reveals that the player bought an edge-case roll, and
   offers `starter_buyback_offer` gold.
3. Give two clear choices:
   - **Sell it back** — permanent, pays the shown gold and returns the exact item
     to Sal.
   - **Keep it** — player retains the item and permanently declines this special
     offer.
4. Persist the pending offer before dialogue so closing/reloading cannot lose,
   duplicate, or exploit it. On return to Sal, resume unresolved dialogue.

If the player sells it back:

- validate that the exact instance is still owned and has not been packed,
  traded, sold, or otherwise mutated during the blocking choice;
- remove it from owned state;
- credit exactly 1.25× pawn quote;
- place it back into the original just-vacated shelf slot;
- set listing mode `mocking_relist` with asking price 10× pawn quote;
- mark the buyback resolved so it never fires again;
- continue Sal's dialogue with an authored mocking line that makes clear he
  profited from the player's ignorance.

If the player keeps it, mark the special sequence resolved and never offer the
special 1.25× buyback again. Buying a later 10× relist also never triggers
another special buyback.

The dialogue's purpose is to teach through action that visually ordinary items
can be valuable because of edge-case floats and that Sal understands this market
better than the player.

## Economy and safety invariants

- Meta gold is the only shelf currency.
- Normal buy→immediate sell always loses gold because resale is 1.5× quote and
  Sal pays 1× quote.
- The starter sequence intentionally grants a one-time arbitrage lesson: buy at
  0.75×, optionally sell at 1.25×, after which Sal asks 10×.
- No transaction duplicates an instance, gold, or slot.
- Replacements destroy only Sal-owned shelf listings; they never touch player
  ownership.
- Shelf inventory never affects run bankroll, run inventory, bag rewards, meta
  collection completion counts before purchase, Steam-facing ownership, housing
  storage, or trade-up eligibility.
- Quote functions are pure and deterministic.
