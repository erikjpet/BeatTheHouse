# 0.5 Inventory and Item Interaction UI Renovation

Status: PLANNED, owner-directed (2026-07-20).

## Intent

Replace the game's menu-like inventory and bag interfaces with a spatial,
container-shaped item surface. When the player owns or opens a Bag, Backpack,
Suitcase, or Trunk, the interface must look like the inside of that specific
container and expose exactly the number of usable spaces that its authoritative
capacity allows. Item selection should use the same interaction language as the
world map and environment object selection: point at an object, commit a
selection, visibly emphasize it, and shift the contextual detail/action focus to
that object.

The same language must follow the player into item transactions. Talking to a
merchant, Sal, a pawn counter, a repair service, home storage, or the trade-up
station should not replace the player's possessions with a text list. The
player visibly pulls out their carried container, selects the item itself, sees
the context-specific quote or action, and confirms through the existing
authoritative transaction path.

This is a UI and interaction renovation. It does not invent a second inventory,
change item effects, change container capacity, change pricing, or move mutation
authority out of `RunState`, `RunActionService`, and `MetaCollectionService`.

## Owner requirements

1. Inventory and bag screens must stop feeling like ordinary menus.
2. All player-owned items must be represented in the inventory experience.
3. The visible inventory must match its container type.
4. A Bag looks like an opened bag; a Trunk looks like an opened trunk; the
   intermediate container types receive their own appropriate interiors.
5. The visual container exposes an appropriate number of item spaces.
6. During a sale or other item interaction, the player pulls out the same bag or
   container and chooses an item from it.
7. Selection, highlight, and focus behavior should feel like the existing map
   popup and environment selection mechanisms.

## Implementation interpretation

The following resolves ambiguities without changing economy rules:

- Use existing item data as capacity authority: Bag 3, Backpack 5, Suitcase 7,
  and Trunk 10. Never create a UI-only capacity table that can drift.
- Preserve the two existing inventory domains:
  - run items use the current `(item id, storage source)` selection identity;
  - meta collection items and unopened bags use their persistent `instance_id`.
- Do not silently assign new gameplay meaning to the visual location of an item.
  Where existing state records a real container, render that assignment. Where
  the current meta loadout is a flat list partitioned by
  `carried_container_rows()`, render that authoritative projection. Reordering
  slots is not persistence unless a separate owner-approved inventory-ordering
  feature is added later.
- A legacy/no-container run uses a clearly distinct "loose carry" presentation
  such as a coat lining or counter mat. It must not impose a new capacity limit.
- Selection moves the visual and informational focus, but does not mutate,
  re-sort, reframe, or make the container jump. The selected item may receive a
  focus ring, local lift, spotlight, and detail-panel transition while the
  container bounds remain stable.
- All current items must be reachable. Remove arbitrary UI truncation such as
  the current first-10 owned-items and first-12 sale-options limits. Multiple
  containers, pages, or stable scrolling are allowed, but every eligible item
  must be selectable.

## Existing seams to preserve

### Run inventory

- `scripts/ui/run_inventory_screen.gd` is a pure code-built UI component. It
  renders a model and emits intent signals.
- `scripts/ui/run_inventory_view_model.gd` builds the model from `RunState` and
  `RunActionService`.
- `scripts/ui/foundation_main.gd` owns public entry points, action execution,
  autosave, messages, and the QA snapshot.
- Current modes are inspect, merchant sale, pawn counter, place container, and
  home container.
- Selection is deliberately keyed by `(id, storage_source)`, because a home
  container view can contain carried and stored variants.

### Run/home containers

- `data/items/items.json` owns the four container definitions and capacities.
- `RunState.place_home_container()` creates a real placed container with a
  stable ID, type, capacity, and stored item IDs.
- `RunState.transfer_item_to_home_container()` and
  `transfer_item_from_home_container()` remain the only run-side transfer
  authority.
- Meta loadout containers mirrored into a run home are read-only there.

### Meta collection and loadout

- `MetaCollectionService` owns exact collection instances, unopened bag
  instances, owned containers, carry capacity, housing storage, loadout,
  open-bag mutation, trade-up tokens, sale tokens, and persistence.
- `carried_container_rows()` is the current authoritative projection of the
  flat loadout into owned containers.
- `MetaSessionController` builds owned-item, bag, sale, shelf, and trade-up rows.
- `foundation_main.gd` currently renders meta container, bag opening, Sal sale,
  and trade-up as generic action-card popups. Those are the primary menu-like
  surfaces to replace.
- The planned Sal resale shelf is a room surface for Sal-owned stock. It stays
  spatial. Only the player's side of a sell/buyback interaction opens the
  carried-container surface.

### Selection reference

- `world_map_overlay_controller.gd`, `world_map_canvas.gd`,
  `environment_interaction_controller.gd`, and
  `environment_interaction_view_model.gd` provide the interaction vocabulary:
  hover/focus, committed selection, a visible selected state, context detail,
  confirm, and input-mode parity.
- Reuse the vocabulary, not map travel semantics. Inventory selection must not
  call travel, consume RNG, or mutate the world.

## Target experience

### Shared spatial container surface

Create one reusable container surface used by both run and meta item screens.
It should receive a pure display model and emit selection/confirm intents. It
must not read or write gameplay state.

Each rendered container has:

- a stable container key and item/container type;
- new open-interior artwork;
- capacity from the authoritative model;
- authored normalized slot anchors matching that art;
- visible empty spaces;
- exact item identity per occupied space;
- selected, focused, hovered, disabled, armed, and multi-selected states;
- count/capacity label and container name;
- accessible labels for all visual-only information.

The likely code boundary is a reusable `InventoryContainerSurface` hosted by the
existing `RunInventoryScreen` and a new meta item-interaction screen. The worker
may choose a different name after auditing, but must not duplicate slot layout,
selection, input routing, or art lookup between run and meta implementations.

### Art contract

Create new pixel-art interiors rather than enlarging the existing 32x32 closed
item icons:

- open bag interior, 3 authored spaces;
- open backpack interior, 5 authored spaces;
- open suitcase interior, 7 authored spaces;
- open trunk interior, 10 authored spaces;
- loose-carry fallback with no implied gameplay capacity;
- optional home-storage treatment for housing capacity that is visually
  distinct from carried containers.

Store visual definitions in one data-driven catalog containing artwork paths,
normalized slot rectangles/centers, safe item-icon scale, label anchor, and
optional foreground occlusion texture. This allows the bag lip, suitcase rim,
or trunk edge to sit in front of item art without hard-coding coordinates in
Main. Use nearest-neighbor pixel rendering and the existing palette/high-
contrast policy.

Art should read at 1280x720 and remain legible at the small-screen test size.
It may use a background, interior, and foreground/occlusion layer, but must not
fake occupied items into the base image.

### Slot behavior

- Empty spaces remain visible and intentionally shaped.
- An item icon occupies one space and keeps aspect ratio.
- Hover is a preview only. It may show a lightweight name/tooltip but does not
  replace the committed selection.
- Select commits the item, draws a strong high-contrast frame, subtly lifts or
  spotlights it, and updates the details/actions.
- The stable container view never pans or changes bounds just because selection
  changes.
- When an action removes an item, select the nearest remaining item by slot
  geometry; if none remains, move focus to Back/Close or the other container.
- When an action adds an item, reveal its destination space and focus it when
  that is the result the player needs to inspect.
- Disabled items stay inspectable and explain why their context action is
  unavailable.
- No hover, focus, selection, open, close, resize, or redraw consumes RNG or
  changes inventory state.

### Input contract

- Mouse/touch: point, hover where supported, click/tap to select, explicit
  action button or second confirm to act.
- Keyboard/controller: directional navigation follows physical slot geometry;
  confirm selects/advances; cancel backs out one layer; shoulders or documented
  equivalent cycle containers/pages.
- Initial focus is deterministic: prior valid selection, otherwise first
  actionable occupied slot, otherwise first inspectable occupied slot,
  otherwise Back/Close.
- Focus is never trapped and never falls behind the overlay.
- Opening the surface blocks environment movement and underlying interactions.
- Destructive or permanent actions keep the existing arm/confirm step. A single
  selection click never sells, pawns, or consumes an item.
- Reduced-motion mode removes lift/zoom/tween movement while retaining the
  selected outline and detail change.

## Required contexts

### 1. Inspect carried inventory

Opening Inventory presents every carried container. A compact container switcher
may show closed silhouettes, but the active content area shows the selected
container open. Items occupy its spaces. Inspecting supports current detail,
attribute badges, active-item selection, repair availability, and sale hints.
If current state supplies loose run items outside a container assignment, show
them on the loose-carry surface without inventing capacity.

### 2. Merchant sale and repair

Interacting with a merchant keeps the environment as context and opens the
player's container as though placed on the counter. The selected item drives the
quote, repair cost, eligibility, and action buttons. Unsellable items remain
visible but clearly disabled. Confirm continues through
`RunActionService.sell_inventory_item()` or `repair_inventory_item()` and the
existing save/message path.

### 3. Run pawn counter

The carried-item side uses the same spatial surface. Pawn tickets and lender
claims that are not literal carried items may occupy a distinct counter tray,
not pretend to be inside the bag. Pawn, cash-ticket, and redeem actions retain
their current service rules and lender IDs.

### 4. Placing and managing home containers

- Placing a carried Bag/Backpack/Suitcase/Trunk shows the selectable carried
  container object and confirms placement through existing `RunState` authority.
- Opening a placed home container shows its actual interior and exact capacity.
- Transfer mode uses a two-surface composition: carried/loose items on one side,
  the opened destination container on the other. Selection plus an explicit
  Store/Take action moves the item; no drag operation bypasses validation.
- Full state is visible before confirmation, and failure messages leave the
  selection intact.
- Meta-loadout mirrors are clearly labeled read-only in the run home.

### 5. Meta home loadout and storage

Replace the current action-card list with a spatial loadout view:

- owned carry containers are individually selectable and show the projected
  packed items from `carried_container_rows()`;
- stored collection items remain accessible through a home-storage surface;
- Pack and Unpack continue through `MetaCollectionService` and respect housing,
  loadout eligibility, and capacity;
- Players Cards, chip stacks, special classes, floats, tier, condition, and
  disabled reasons remain visible;
- all owned instances are reachable, including more than ten items;
- homeless/back-alley behavior remains authoritative and gets an honest visual
  fallback rather than a fake finite bag.

### 6. Unopened collection bags and reveal

Unopened reward bags are exact persistent instances. Selecting one highlights
the bag object and offers Open. Capacity must be checked by the existing service
before any reveal animation. On success:

1. the selected bag opens from its current visual position;
2. the existing seeded service roll happens exactly once;
3. the exact resulting item is revealed with tier/floats/condition;
4. the item appears in the appropriate storage/carry surface after refresh;
5. save and result messaging retain current semantics.

Closing, reopening, hovering, or replaying animation must never reroll or grant
again. On failure, the bag remains and the reason is shown.

### 7. Sal and other meta sales

When selling to Sal, the player's carried/storage surface opens over the pawn
counter context. Collection items and unopened bags remain distinct visual
objects with exact instance IDs. Selecting one requests its current quote;
confirm uses the existing arm/confirm token transaction. The planned float-
rarity pricing and special starter buyback are consumed, not recomputed in UI.
Sal's dialogue can react while the selected container stays visible.

Buying Sal-owned shelf stock still begins from the physical six-slot shelf. The
inventory surface may open afterward to show the exact purchased item arriving,
but it must not convert Sal's room back into a shop list.

### 8. Trade-up selection

Trade-up becomes a multi-select container interaction instead of choosing a
precomputed first-five card:

- player selects five exact eligible instances;
- selected inputs get a numbered/checkmarked state that coexists with current
  focus;
- the UI explains same-collection and same-tier requirements;
- invalid items remain inspectable and explain incompatibility;
- a summary tray shows all five exact inputs and the possible next tier;
- Confirm first arms the service transaction, then explicitly confirms the
  permanent consumption;
- result reveal focuses the new exact instance;
- canceling or closing clears only ephemeral UI selection, not items.

The service remains final authority and must revalidate at arm and confirm.

## View-model contract

The shared surface should consume a normalized model equivalent to:

```gdscript
{
	"mode": "inspect",
	"context": {"kind": "run", "actor_id": "", "container_id": ""},
	"title": "Inventory",
	"summary": "...",
	"containers": [
		{
			"key": "meta_bag_01",
			"container_type": "bag",
			"display_name": "Bag",
			"capacity": 3,
			"read_only": false,
			"slots": [
				{
					"slot_index": 0,
					"occupied": true,
					"selection_key": "meta_item:1042",
					"item": {},
					"actionable": true,
					"disabled_reason": ""
				}
			]
		}
	],
	"selected_key": "meta_item:1042",
	"multi_selected_keys": [],
	"detail": {},
	"actions": [],
	"layout": {"presentation": "single", "stable_view": true}
}
```

Selection keys must be domain-safe and stable:

- run carried/stored: include item ID and storage source;
- meta item: include exact item instance ID;
- unopened bag: include exact bag instance ID;
- pawn ticket/debt: include its stable debt/ticket ID;
- empty slot: never masquerades as an item selection.

The renderer may not infer prices, eligibility, floats, item effects, container
capacity, or destructive action validity. Those arrive in the model.

## Architecture and mutation boundary

1. Add a data-driven container visual catalog and validation loader.
2. Add a pure shared spatial surface for art, slot geometry, focus, pointer hit
   testing, and selection signals.
3. Renovate `RunInventoryScreen` to host that surface while preserving its
   public API and legacy signals/snapshot compatibility.
4. Extend `RunInventoryViewModel` to provide container/slot models without
   moving service rules into UI.
5. Add a focused meta item-interaction view model and host component. It may
   share the full screen shell or only the spatial surface, but must not render
   another generic action-card inventory.
6. Keep `foundation_main.gd` as coordinator: open/close, environment blocking,
   service mutation, save, dialogue, refresh, and legacy entry-point wrappers.
7. Keep all tokenized destructive flows tokenized. UI state never stands in for
   service validation.
8. Preserve unknown save fields and legacy profiles. This renovation should not
   require a save schema change unless persistent slot ordering is explicitly
   added, which is out of scope.

## Responsive composition

- At 1280x720, use a large container stage plus an item detail/action panel.
- In transfer mode, two container stages may share the width if slot targets
  remain legible; otherwise use an explicit Source/Destination toggle.
- At small widths, stack container above details and keep selected item visible.
- Never shrink input targets below the current small-screen policy.
- Container switching/paging must be explicit and show position/count.
- The layout may scroll details, but it should not scroll the selected container
  out of view when an action is being confirmed.
- Background environment remains recognizable but dimmed; transaction context
  (merchant, Sal, home container) stays labeled.

## Error and recovery behavior

- If visual catalog data is missing or corrupt, render a neutral container
  frame with generated slot geometry; do not hide items.
- If an art texture is missing, show a visible placeholder and preserve all
  interactions.
- If state changes while open, rebuild from authority and reconcile selection by
  stable key.
- If the selected item disappeared, choose nearest valid neighbor and cancel any
  now-invalid armed UI state.
- If an arm/confirm token expires, show the service message and refresh; never
  retry automatically.
- Duplicate item display names never collapse exact instances.
- Full containers, no-capacity states, meta-only items, read-only mirrored
  containers, and unsellable items always explain the reason.

## Testing requirements

### Pure/model tests

- Catalog covers bag/backpack/suitcase/trunk with exactly 3/5/7/10 valid,
  non-overlapping, in-bounds slot anchors.
- Capacity is sourced from item definitions and mismatches fail validation.
- Domain-safe selection keys distinguish same ID/different source and duplicate
  meta definitions/different instances.
- Every owned item and unopened bag appears exactly once in its applicable view.
- Projection across multiple containers is deterministic and consumes no RNG.
- Trade-up multi-selection accepts exactly five compatible instances and
  rejects invalid sets before service arm without replacing service validation.

### Component tests

- Open each container type and verify art, empty spaces, occupied spaces, count,
  selection, details, and actions.
- Mouse, keyboard, controller, and touch-equivalent paths select the same stable
  item.
- Spatial navigation follows nearest slot geometry.
- Hover does not replace committed selection.
- Selection changes do not move or resize container bounds.
- Reduced motion and high contrast preserve state readability.
- Removal selects nearest neighbor; addition focuses reveal destination.
- Transfer mode correctly distinguishes carried and stored versions.
- More than 10/12 items remain reachable.
- Small-screen layout stays on-screen with valid target sizes.

### Integration tests

- Inspect and set active run item.
- Sell and repair through the run merchant.
- Pawn, cash tickets, and redeem at the run pawn counter.
- Place each container type and store/take at capacity boundaries.
- Meta pack/unpack with multiple owned containers and housing tiers.
- Open a bag once, round-trip save, and prove no reroll on UI reopen.
- Sell exact item and exact unopened bag to Sal using arm/confirm tokens.
- Complete and cancel a five-item trade-up.
- Planned Sal starter purchase/buyback dialogue continues with the inventory
  surface open and transfers the exact instance.
- Environment input stays blocked while the surface is open and resumes after
  close.
- Existing run inventory QA snapshot remains backward compatible.

### Performance and determinism

- Opening and changing selection do not roll RNG or mutate saves.
- Texture and layout catalogs are cached.
- Avoid clearing/recreating the whole slot tree on hover or focus changes; pool
  controls or update state in place.
- Repeated open/close does not leak controls, signals, tweens, or focus owners.
- Add a bounded stress fixture with the largest supported owned collection and
  record render/control counts and selection latency.

## Delivery phases

1. Audit and migration map; run baseline validation.
2. Container art catalog, new interior assets, catalog validation, pure spatial
   surface, and component tests.
3. Run inventory migration with existing API/snapshot compatibility.
4. Run merchant, pawn, placement, and home-container transfer contexts.
5. Meta home loadout/storage and unopened-bag reveal.
6. Sal sale/buyback integration and trade-up multi-selection.
7. Accessibility, small-screen, deterministic refresh, stress, and full
   regression pass.

Each phase must leave the project compiling. Do not postpone all integration to
the final phase.

## Acceptance criteria

The renovation is complete only when:

- inventory no longer presents owned items as a generic text/card list;
- each Bag/Backpack/Suitcase/Trunk visibly has its own opened interior and exact
  3/5/7/10 spaces;
- all applicable owned items are reachable, regardless of current row limits;
- selection visibly focuses the item and changes context details without moving
  the container;
- the same spatial selection language works for inspect, merchant, pawn, home
  storage, meta loadout, unopened bags, Sal sales, and trade-up;
- all mutations still pass through existing authoritative services and token
  flows;
- UI operations consume no RNG and require no save migration;
- mouse, keyboard, controller, touch, reduced-motion, high-contrast, 1280x720,
  and small-screen paths are covered;
- existing unrelated gameplay and the physical Sal shelf remain intact;
- validation and relevant Godot suites pass without weakening existing tests.
